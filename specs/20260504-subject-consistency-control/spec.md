# Feature Specification: 主体一致性控制

**Feature**: `subject-consistency-control`
**Created**: 2026-05-04
**Status**: Draft
**Input**: User description: "视频生成过程需要保证人物主体的一致性"

## 用户需求概述

用户希望在视频生成过程中，同一人物主体在不同镜头中保持视觉特征的一致性。具体需求：
- **特征一致性**：允许同一主体有不同表情/角度/服装，但保持核心视觉特征可识别为同一人物
- **支持变体**：同一人物可以有不同表情、动作、服装，但仍是同一人物
- **手动确认更新**：允许重新生成主体图，但需用户明确确认，并更新版本号

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 主体特征锁定 (Priority: P1)

用户在生成分镜后，进入主体生成阶段。系统为每个主体名称（如"主角"、"配角A"）生成一张基础图片，并自动提取该主体的视觉特征描述。用户可以编辑这些特征描述来"锁定"主体的核心视觉属性（如发色、服装风格、面部特征等）。一旦锁定，后续生成该主体的变体时，系统会参考这些特征描述，确保不同变体仍可识别为同一人物。

**Why this priority**: 这是主体一致性的核心能力。没有特征锁定，就无法保证后续生成的变体与原始主体视觉一致。这是整个功能的基础。

**Technical Implementation**:

1. **数据模型扩展**：
   - 在 `subjects` 表中新增 `feature_description` 字段（TEXT），存储主体的视觉特征描述
   - 新增 `base_subject_id` 字段，用于关联变体与基础主体
   - 新增 `variant_type` 字段，标识主体类型（`base`/`variant`）

2. **特征提取流程**：
   - 当 AI 生成主体图后，调用文本模型提取视觉特征
   - 特征描述包括：性别、年龄、发色、发型、肤色、体型、服装风格等
   - 用户可以在 UI 中编辑这些描述

3. **API 变更**：
   - `POST /api/projects/{projectId}/subjects:generate` 返回时包含 `feature_description`
   - `PUT /api/projects/{projectId}/subjects/{subjectId}` 支持更新 `feature_description`

4. **前端实现**：
   - 在主体资产页面显示特征描述编辑框
   - 提供"锁定特征"按钮，点击后特征描述进入只读状态
   - 锁定后显示"解锁并重新生成"选项

**Independent Test**: 可以通过创建一个项目，生成主体后编辑特征描述，然后验证特征描述是否正确保存和显示来独立测试。

**Acceptance Scenarios**:

1. **Given** 项目已生成分镜脚本且包含主体引用，**When** 用户点击"生成主体图"，**Then** 系统生成主体图片并自动提取特征描述，显示在 UI 中
2. **Given** 主体已生成且显示特征描述，**When** 用户编辑特征描述并保存，**Then** 特征描述更新成功，后续变体生成使用更新后的描述
3. **Given** 主体特征已编辑，**When** 用户点击"锁定特征"，**Then** 特征描述变为只读，UI 显示锁定状态

---

### User Story 2 - 主体变体生成 (Priority: P2)

用户在渲染阶段，可以为特定镜头生成主体的变体图片。变体保持基础主体的核心特征，但可以有不同的表情、动作或服装。系统在生成变体时，会将基础主体的特征描述作为参考，确保变体与基础主体在视觉上保持可识别的一致性。

**Why this priority**: 变体生成是用户需求的"支持变体"部分，它依赖 US1 的特征锁定功能。有变体能力后，用户可以为不同镜头定制主体状态，同时保持人物识别一致性。

**Technical Implementation**:

1. **变体生成 API**：
   - 扩展 `POST /api/projects/{projectId}/subjects/shots/{shotId}:generate` 接口
   - 新增 `variant_hint` 参数，指定变体方向（如"微笑表情"、"运动装束"、"侧面角度"）
   - 生成时将 `feature_description` + `variant_hint` 组合作为 prompt

2. **Prompt 构建策略**：
   ```
   基础特征：{feature_description}
   变体要求：{variant_hint}
   风格要求：{style_prompt}
   ```

3. **数据存储**：
   - 变体主体的 `base_subject_id` 指向基础主体
   - `variant_type` = 'variant'
   - `profile` 字段记录变体说明

4. **前端实现**：
   - 在镜头主体图面板中新增"生成变体"按钮
   - 提供变体方向选择器（预设选项：表情变化、服装变化、动作变化、视角变化）
   - 或允许用户输入自定义变体描述

**Independent Test**: 可以在已生成基础主体的项目中，为某个镜头生成变体，验证变体图片是否保持与基础主体的视觉相似性。

**Acceptance Scenarios**:

1. **Given** 项目已有锁定的基础主体，**When** 用户在镜头 X 点击"生成变体"并选择"微笑表情"，**Then** 系统生成该镜头的变体主体图，变体保持基础特征但有微笑表情
2. **Given** 镜头已生成变体主体图，**When** 用户查看主体列表，**Then** 显示变体与基础主体的关联关系
3. **Given** 基础主体特征已更新，**When** 用户重新生成变体，**Then** 新变体使用更新后的特征描述

---

### User Story 3 - 主体更新确认流程 (Priority: P3)

当用户希望更新主体图时（重新生成或上传新图片），系统会提示确认。确认后，系统会更新版本号，并可选择是否级联更新所有使用该主体的镜头渲染任务。历史版本会保留，用户可以回退到之前的版本。

**Why this priority**: 版本管理是用户需求的"手动确认更新"部分。它提供安全网，允许用户在更新不满意时回退，是完善的功能补充。

**Technical Implementation**:

1. **版本管理机制**：
   - 利用现有 `image_version` 字段
   - 新增 `subject_versions` 表记录历史版本：
     - `subject_id`: 关联主体
     - `version`: 版本号
     - `image_path`: 历史图片路径
     - `feature_description`: 历史特征描述
     - `created_at`: 创建时间

2. **确认流程**：
   - 用户点击"重新生成"时，弹出确认对话框
   - 对话框显示：当前版本、即将创建的新版本号、影响范围（多少镜头使用此主体）
   - 用户可选择：仅更新主体图 / 更新主体图并重新渲染相关镜头

3. **API 变更**：
   - `POST /api/projects/{projectId}/subjects/{subjectId}:regenerate` 新增确认接口
   - 请求体：`{ "cascade_render": boolean }`
   - 返回：新版本主体信息

4. **回退功能**：
   - `POST /api/projects/{projectId}/subjects/{subjectId}:rollback`
   - 参数：`{ "target_version": int }`

5. **前端实现**：
   - 主体资产卡片显示版本号和历史记录按钮
   - 点击历史记录显示版本列表，可选择回退
   - 重新生成按钮带确认对话框

**Independent Test**: 可以在已生成主体的项目中，触发重新生成流程，验证确认对话框显示正确，版本号递增，历史版本可回退。

**Acceptance Scenarios**:

1. **Given** 主体已锁定且版本为 1，**When** 用户点击"重新生成"并确认，**Then** 系统创建版本 2，版本 1 保留在历史记录中
2. **Given** 用户已更新主体到版本 2，**When** 用户点击"回退到版本 1"，**Then** 当前版本恢复为版本 1 的图片和特征
3. **Given** 重新生成时用户选择"级联渲染"，**When** 主体更新完成，**Then** 使用该主体的镜头自动触发重新渲染

---

### Edge Cases

- 当基础主体被删除时，已生成的变体如何处理？→ 变体保留但失去基础关联，`base_subject_id` 置空
- 当特征描述过长时如何处理？→ 限制 1000 字符，超出时截断并提示用户
- 当变体生成失败时如何处理？→ 回退使用基础主体图，记录失败原因
- 当用户在未锁定特征时生成变体如何处理？→ 使用当前特征描述，但提示用户建议先锁定
- 多个镜头同时请求生成变体时如何处理？→ 并行生成，各自独立

## 架构约束

**场景与镜头关系**：用户反馈"场景与镜头一般并不会一一对应"。本功能设计已考虑此约束：
- 主体关联到**镜头层面**（通过 `shot_id`），而非场景层面
- 支持一个场景对应多个镜头的灵活编排
- 渲染时按镜头选择主体，不受场景-镜头映射限制

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在生成主体图后自动提取视觉特征描述
- **FR-002**: 用户 MUST 能够编辑和锁定主体的特征描述
- **FR-003**: 系统 MUST 在生成变体时参考基础主体的特征描述
- **FR-004**: 用户 MUST 能够为特定镜头生成主体变体
- **FR-005**: 系统 MUST 在重新生成主体前要求用户确认
- **FR-006**: 系统 MUST 保留主体的历史版本
- **FR-007**: 用户 MUST 能够回退到历史版本
- **FR-008**: 系统 MUST 在更新主体时提供级联渲染选项
- **FR-009**: 变体主体 MUST 保持与基础主体的关联关系
- **FR-010**: 渲染时 MUST 优先使用镜头专属变体，其次全局主体

### Key Entities

- **Subject（主体）**: 表示视频中的角色/人物，包含特征描述、版本、类型（基础/变体）、关联关系
- **SubjectVersion（主体版本）**: 表示主体的历史版本记录，包含图片路径、特征描述、创建时间
- **FeatureDescription（特征描述）**: 主体的视觉特征文本描述，用于保证生成一致性
- **VariantHint（变体提示）**: 生成变体时的方向指引，如表情、服装、动作等变化方向

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户生成的同一主体在不同镜头中，视觉特征相似度达到可识别程度（用户主观评价）
- **SC-002**: 用户可以在 30 秒内完成主体特征的编辑和锁定
- **SC-003**: 用户可以在 1 分钟内为镜头生成主体变体
- **SC-004**: 版本回退操作在 10 秒内完成
- **SC-005**: 90% 的用户在首次使用后理解特征锁定的作用
