## User Scenarios & Testing *(mandatory)*

### User Story 1 - 智能生成可审核的创意方案 (Priority: P1)

内容创作者进入平台后，可以通过自然语言输入视频创意需求，系统基于输入自动完成智能场景设计，并在进入后续制作前提供明确的人工审核环节。该阶段输出的是“场景方案”，用于定义视频主题、风格、角色、场景、叙事节奏与目标成片方向。用户需要能够在系统生成后发起 review，对方案进行确认、退回或重新生成，避免错误创意进入下游制作链路。

**Why this priority**: 没有稳定的创意输入与人工确认，后续分镜、素材和视频生成都会偏离目标，因此这是整个平台最关键的入口能力。

**Technical Implementation**:

- 产品定位：建设一个智能视频生成平台，提供从创意输入到最终视频输出的完整闭环。
- 技术约束：前端使用 Node.js 技术栈，后端使用 Python 技术栈。
- 入口表单至少包含：视频主题、目标受众、风格偏好、时长预期、角色设定、语言、补充说明。
- 系统基于用户输入生成结构化场景设计结果，至少包含：创意摘要、视频目标、场景列表、角色与主体说明、视觉风格建议、旁白/字幕基调建议。
- 平台提供场景设计 review 流程，状态包括：`scene_generated`、`scene_reviewing`、`scene_approved`、`failed`。
- review 页展示原始输入、生成结果、版本号、更新时间和操作记录。
- 当用户驳回时，系统保留旧版本并允许重新生成新版本。
- 后端负责生成与状态流转，前端负责输入、展示、review 和版本切换。

**Independent Test**: 输入一段明确的视频需求后，系统能生成一版场景设计，并允许用户完成“通过 / 驳回 / 重新生成”的独立闭环。

**Acceptance Scenarios**:

1. **Given** 用户填写视频需求并提交，**When** 系统完成生成，**Then** 页面展示结构化场景设计结果并进入待 review 状态。
2. **Given** 用户正在查看一版场景设计，**When** 用户点击驳回并填写修改意见，**Then** 系统保留当前版本并允许重新生成新版本。
3. **Given** 用户已确认场景方案符合预期，**When** 用户点击通过，**Then** 系统将场景设计状态更新为已通过，并允许进入分镜脚本阶段。

### User Story 2 - 生成并编辑分镜头脚本 (Priority: P1)

当场景设计 review 通过后，平台需要自动进行智能分镜头脚本设计。每个分镜头脚本必须包含字幕和配音信息，并支持配置分镜头最大时长，默认值为 10 秒。生成完成后，用户需要再次 review；在此阶段，用户不仅能审批，还能对单个分镜脚本内容进行编辑，以确保镜头内容、台词、节奏与视觉动作符合预期。

**Why this priority**: 分镜脚本直接决定视频生成质量，是创意设计与视频生成之间的核心桥梁。

**Technical Implementation**:

- 系统在场景设计批准后自动触发分镜生成任务。
- 每个分镜必须包含：镜头编号、镜头描述、时长、字幕、配音文案、镜头顺序、关联主体。
- 支持配置单个分镜头最大时长，默认 `10s`。
- 分镜结果必须按顺序组织，保证后续视频拼接顺序一致。
- 分镜 review 编辑器支持：编辑字幕、编辑配音、调整时长、修改镜头描述、调整顺序、删除镜头、新增镜头。
- 编辑后系统重新校验总时长、镜头顺序和必填字段完整性。
- 分镜状态包括：`storyboard_generated`、`storyboard_reviewing`、`storyboard_approved`、`failed`。
- 后端支持分镜版本管理，前端提供时间线 + 卡片式镜头编辑视图。

**Independent Test**: 在已有场景设计的前提下，系统能生成完整分镜列表，用户能调整其中任一镜头的字幕、配音与时长，并提交通过。

**Acceptance Scenarios**:

1. **Given** 场景设计已经审批通过，**When** 系统执行分镜生成，**Then** 生成的每个分镜都包含字幕、配音和不超过默认 10 秒的时长。
2. **Given** 用户在 review 分镜脚本，**When** 用户修改某个镜头的字幕与配音，**Then** 系统保存修改结果并标记为最新脚本版本。
3. **Given** 某个镜头时长超出允许上限，**When** 用户尝试保存，**Then** 系统阻止提交并提示超出最大分镜时长限制。

### User Story 3 - 统一主体素材并行生成视频 (Priority: P1)

分镜脚本 review 通过后，平台需要先抽取所有镜头中涉及的人物主体，并统一生成图片素材，以保证跨镜头角色形象一致。之后系统基于镜头脚本和对应图片，对每个镜头并行执行视频生成。生成结果支持缓存到预先配置的目录，以便复用和调试。最后，系统按照分镜顺序对各镜头视频进行拼接，返回并存储最终成片。

**Why this priority**: 这是平台能否真正产出成片的核心链路。

**Technical Implementation**:

- 从全部分镜脚本中识别人物主体，合并重复主体，形成统一主体资产清单。
- 每个主体必须具有唯一标识、角色设定、外观描述和参考图生成参数。
- 对每个主体统一生成形象图，并关联到相关镜头。
- 以单分镜为最小任务单元并行执行视频生成。
- 每个视频生成任务输入包括：镜头脚本、字幕、配音、主体图片、风格参数、模型参数。
- 任务状态包括：`queued`、`running`、`succeeded`、`failed`。
- 支持将生成的视频缓存到预配置目录。
- 缓存键由镜头脚本内容哈希、主体图版本、模型版本和渲染参数共同组成。
- 全部镜头完成后，系统根据分镜顺序拼接视频，并保持字幕、配音与片段时长对齐。
- 最终视频需存储项目 ID、版本号、文件路径、总时长、生成时间。

**Independent Test**: 在一组已审批分镜的基础上，系统能自动完成主体抽取、统一出图、并行生成镜头视频、缓存结果并输出最终拼接视频。

**Acceptance Scenarios**:

1. **Given** 分镜脚本已审批通过，**When** 系统抽取人物主体，**Then** 相同主体在不同镜头中只生成一套统一形象图并复用。
2. **Given** 存在多个待生成镜头，**When** 平台启动视频生成，**Then** 系统并行执行各镜头任务并分别记录状态。
3. **Given** 某个镜头存在可复用缓存，**When** 用户再次发起生成，**Then** 系统优先使用缓存目录中的视频结果而不是重复生成。
4. **Given** 所有镜头已生成完成，**When** 系统执行拼接，**Then** 返回一个按分镜顺序合成的最终视频并完成存储。

### User Story 4 - 管理项目进度与最终交付 (Priority: P2)

为了让用户可持续使用平台，系统需要以“视频项目”为组织单位，管理从创意输入、场景设计、分镜审核、主体出图、镜头生成到最终拼接的全过程，并清晰展示当前进度与结果。用户应能随时查看某一项目卡在哪个阶段、哪些镜头正在生成、最终视频是否可下载或继续迭代。

**Why this priority**: 这是提升产品可用性与可运营性的关键能力，保证复杂异步链路对用户透明。

**Technical Implementation**:

- 系统以“项目”作为顶层实体，串联场景、分镜、主体、视频片段和最终成片。
- 项目首页展示：项目名称、状态、最近更新时间、阶段进度、失败重试入口。
- 统一项目阶段包括：`idea_submitted`、`scene_generated`、`scene_reviewing`、`scene_approved`、`storyboard_generated`、`storyboard_reviewing`、`storyboard_approved`、`subject_generating`、`subject_ready`、`video_rendering`、`video_render_partial_failed`、`compositing`、`completed`、`failed`。
- 若单个镜头生成失败，系统支持单镜头重试，而不是整体重跑。
- 最终交付页支持在线播放、下载、查看生成参数和再次发起新版本生成。

**Independent Test**: 创建一个项目并完成一次完整生产后，用户能在同一项目下查看全部阶段状态、片段结果与最终成片。

**Acceptance Scenarios**:

1. **Given** 用户进入项目详情页，**When** 项目正在执行镜头生成，**Then** 页面展示当前阶段、各镜头状态和总体进度。
2. **Given** 最终视频已拼接完成，**When** 用户打开项目结果页，**Then** 能查看并获取已存储的成片。

### Edge Cases

- 当用户输入过于模糊、无法生成有效场景设计时，系统应提示补充信息而不是直接进入后续流程。
- 当场景设计已通过但分镜生成失败时，系统应保留已审批场景，允许重新触发分镜生成。
- 当用户在分镜 review 中删除了关键主体镜头后，系统应重新计算主体资产清单。
- 当某些镜头视频生成失败而其他镜头成功时，系统应允许局部重试，并避免已成功片段重复生成。
- 当缓存目录不可写或不存在时，系统应记录错误并回退为非缓存生成模式。
- 当最终拼接失败时，系统应保留所有镜头片段和错误日志，供重新拼接或排查。
- 当配音生成成功但镜头视频生成失败时，系统应复用已完成的配音轨，不重复生成音频。
- 当单镜头没有识别出人物主体时，系统应允许以纯文本提示继续生成非角色镜头。

## Supplemental Design Constraints

### Roles & Permissions

- **创作者**：创建项目、提交创意、编辑分镜、发起生成、查看与下载结果。
- **审核者**：审核场景设计、审核分镜脚本、填写驳回意见。
- **管理员**：配置缓存目录、对象存储、模型版本、并发阈值和系统告警。

### Storage Strategy

- **数据库**：存储项目元数据、状态机、审核记录、版本信息、错误日志。
- **缓存目录**：存储临时镜头视频、可复用渲染结果和拼接中间产物。
- **对象存储/文件存储**：存储主体图片、镜头视频、最终成片。
- **日志系统**：存储模型调用日志、任务追踪日志、失败原因与审计轨迹。

### Model Orchestration

1. 场景模型生成结构化场景设计。
2. 分镜模型生成镜头级脚本。
3. 主体抽取器识别角色并去重。
4. 图片模型生成统一主体图。
5. 配音模型生成镜头配音轨。
6. 视频模型按镜头并行生成视频片段。
7. 拼接模块合成字幕、配音与视频，导出最终成片。

### Audio/Video Rules

- 默认输出分辨率为 `1920x1080`。
- 默认帧率为 `25fps`。
- 默认画幅比例为 `16:9`。
- 字幕按镜头维度管理，并必须与镜头时间线对齐。
- 默认配音语言为普通话，可在后续版本扩展多语言。
- 拼接阶段必须保证字幕轨、配音轨和视频轨时间对齐。

### Cache Policy

- 缓存键由 `shot_content_hash + subject_image_version + model_version + render_config` 组成。
- 仅当镜头脚本、主体图版本、模型版本、渲染参数全部一致时才允许命中缓存。
- 任一输入变化、模型升级或缓存损坏时，缓存失效。
- 缓存清理按最近访问时间和磁盘占用阈值执行。

### Recovery Strategy

- 单任务默认最大重试次数为 `3`。
- 超时任务自动标记为失败，并进入人工排查列表。
- 单镜头失败不阻塞其他镜头生成。
- 拼接失败允许在不重新渲染镜头的情况下单独重试拼接。

### API Boundary Outline

- `POST /api/projects`
- `GET /api/projects/{projectId}`
- `POST /api/projects/{projectId}/scene-design:generate`
- `POST /api/projects/{projectId}/scene-design:review`
- `POST /api/projects/{projectId}/storyboard:generate`
- `PUT /api/projects/{projectId}/storyboard/shots/{shotId}`
- `POST /api/projects/{projectId}/storyboard:review`
- `POST /api/projects/{projectId}/subjects:generate`
- `POST /api/projects/{projectId}/renders:start`
- `POST /api/projects/{projectId}/renders/shots/{shotId}:retry`
- `POST /api/projects/{projectId}/final-video:compose`
- `GET /api/projects/{projectId}/final-video`

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须允许用户输入视频创意需求，并触发智能场景设计。
- **FR-002**: 系统必须为场景设计提供 review 流程，支持通过、驳回和重新生成。
- **FR-003**: 系统必须在场景设计通过后自动生成智能分镜头脚本。
- **FR-004**: 系统必须支持配置单个分镜头最大时长，默认值为 `10s`。
- **FR-005**: 系统生成的每个分镜脚本必须包含字幕和配音内容。
- **FR-006**: 系统必须为分镜脚本提供 review 能力，并支持用户编辑分镜内容。
- **FR-007**: 系统必须在分镜审批通过后抽取镜头中涉及的人物主体，并生成统一图片素材。
- **FR-008**: 系统必须支持基于脚本和主体图片并行生成各分镜视频。
- **FR-009**: 系统必须支持将生成的视频缓存到已配置目录。
- **FR-010**: 系统必须在镜头视频完成后，按分镜顺序进行视频拼接。
- **FR-011**: 系统必须返回最终拼接视频，并完成存储。
- **FR-012**: 系统必须以项目维度管理从输入、review、生成到成片的全链路状态。
- **FR-013**: 系统必须支持记录版本信息，至少覆盖场景设计版本、分镜版本和最终视频版本。
- **FR-014**: 前端必须基于 Node.js 技术栈实现交互界面，后端必须基于 Python 技术栈实现智能生成与任务编排。
- **FR-015**: 系统必须区分创作者、审核者、管理员三类角色，并限制其可执行操作范围。
- **FR-016**: 系统必须提供核心项目、场景、分镜、渲染、成片接口，用于前后端联调与任务编排。
- **FR-017**: 系统必须保存主体图片、镜头视频、最终成片及其版本化元数据。
- **FR-018**: 系统必须支持单镜头级别的失败重试和单独拼接重试。
- **FR-019**: 系统必须支持字幕、配音、视频三轨在拼接阶段对齐。
- **FR-020**: 系统必须支持基于缓存键进行渲染结果命中、失效判断与清理。

### Non-Functional Requirements

- **NFR-001**: 除模型生成本身外，页面关键交互响应时间必须小于 `2s`。
- **NFR-002**: 项目状态查询接口平均响应时间必须小于 `500ms`。
- **NFR-003**: 系统必须支持单项目至少 `20` 个镜头并行渲染。
- **NFR-004**: 关键任务状态必须可追踪、可审计、可告警。
- **NFR-005**: 缓存目录不可用时，系统必须自动降级为非缓存模式。
- **NFR-006**: 最终成片与关键元数据必须持久化保存，避免任务完成后结果丢失。
- **NFR-007**: 系统必须支持模型版本管理，便于后续灰度切换与回溯。
- **NFR-008**: 敏感配置、存储密钥和模型访问凭据不得暴露给前端。
- **NFR-009**: 任务失败日志必须包含项目 ID、镜头 ID、模型版本与错误摘要。
- **NFR-010**: 系统必须支持镜头级局部失败而不阻塞其他成功任务的结果复用。

### Key Entities *(include if feature involves data)*

- **视频项目（Project）**: 表示一次完整的视频生成任务，关键属性包括 `id`、`name`、`creatorId`、`status`、`currentStage`、版本号、创建时间、更新时间。
- **场景设计（Scene Design）**: 表示由用户输入推导出的创意与场景方案，关键属性包括 `projectId`、`version`、`inputPrompt`、`sceneSummary`、`sceneList`、`reviewStatus`、`reviewComment`。
- **分镜脚本（Storyboard Shot）**: 表示单个镜头脚本，关键属性包括 `shotId`、`sequence`、`duration`、`description`、`subtitleText`、`voiceoverText`、`subjectRefs`、`status`。
- **主体资产（Subject Asset）**: 表示从分镜中抽取出的角色/人物主体及其统一参考图片，关键属性包括 `subjectId`、`name`、`profile`、`imagePath`、`imageVersion`。
- **镜头视频任务（Shot Render Task）**: 表示单镜头视频生成任务，关键属性包括 `taskId`、`shotId`、`status`、`cacheKey`、`cacheHit`、`renderPath`、`retryCount`、`errorMessage`。
- **最终成片（Final Video）**: 表示拼接后的完整视频，关键属性包括 `videoId`、`projectId`、`version`、`storagePath`、`duration`、`resolution`、`createdAt`。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在一次项目流程中完成“创意输入 → 场景 review → 分镜 review → 视频生成 → 成片获取”的完整闭环，无需跳出系统。
- **SC-002**: 100% 的分镜脚本记录都包含时长、字幕和配音字段。
- **SC-003**: 单项目支持至少 `20` 个镜头级任务并行执行。
- **SC-004**: 在存在相同输入和相同素材版本时，系统能够命中缓存并复用既有镜头视频结果。
- **SC-005**: 用户可以对任一分镜进行编辑并重新提交，而不影响其他镜头的顺序与可生成性。
- **SC-006**: 最终输出视频必须具备可访问的存储记录，并能关联到其来源项目和分镜版本。
- **SC-007**: 单镜头失败时，项目仍可继续推进，并允许对失败镜头单独重试。
- **SC-008**: 项目详情页能够完整展示阶段状态、镜头级进度与失败信息。
