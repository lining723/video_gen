# Compound Output Templates

This document defines the output templates for knowledge documents
generated by compound.

## Common Frontmatter Specification

All compound-generated files must include YAML frontmatter:

```yaml
---
last_compound_scan: YYYY-MM-DD
compound_scan_mode: full|incremental
---
```

- `last_compound_scan`: Date of the last compound scan
- `compound_scan_mode`: Scan mode (full or incremental)

## Common Table of Contents Section

For dimensions with `toc_required: true`, insert a table of contents
section after frontmatter and before the main content headings. TOC
entries are generated from the `output.structure` field in schema.yaml
and **only include sections with actual content** (sections with
required=false are omitted if empty).

```markdown
## Table of Contents

- [Overview](#overview)
- [Creational Patterns](#creational-patterns)
- [Structural Patterns](#structural-patterns)
- [Behavioral Patterns](#behavioral-patterns)
- [Pattern Dependencies](#pattern-dependencies)
- [Source References](#source-references)
```

> Note: TOC entries must exactly match the actual section headings in
> the document. When the null-value passthrough rule is in effect,
> omitted sections do not appear in the TOC.

## Cross-Reference Format

When a dimension's `dedup.secondary_for` declares that a topic should
be detailed by another dimension, use a cross-reference format instead
of duplicating content:

```markdown
> 🔗 See [Topic Name](relative-path#anchor)
```

### Relative Path Rules

Cross-reference paths must be calculated based on the **current file's
directory**, not the project root. All compound-generated files reside
in `docs/`, therefore:

| Current File | Target File | Correct Path | Common Mistake |
|---|---|---|---|
| `docs/CODING.md` | `docs/arch/patterns.md` | `arch/patterns.md` | ~~`docs/arch/patterns.md`~~ |
| `docs/SECURITY.md` | `docs/RELIABILITY.md` | `RELIABILITY.md` | ~~`../RELIABILITY.md`~~ |
| `docs/SECURITY.md` | `docs/arch/patterns.md` | `arch/patterns.md` | ~~`../arch/patterns.md`~~ |
| `docs/arch/index.md` | `docs/CODING.md` | `../CODING.md` | ~~`docs/CODING.md`~~ |
| `docs/references/index.md` | `docs/arch/patterns.md` | `../arch/patterns.md` | ~~`docs/arch/patterns.md`~~ |

**Core Principles**:
- Same-level files under `docs/` referencing each other: use filename
  directly (e.g., `RELIABILITY.md`)
- Files under `docs/` referencing subdirectories: use relative path
  (e.g., `arch/patterns.md`)
- Files under `docs/arch/` or `docs/references/` referencing same-level
  or parent: use `../` to return to `docs/` then navigate
- **Never** use the `docs/` prefix in files within `docs/`

### Anchor Rules

Anchors in cross-references (the part after `#`) **must correspond to
actual headings in the target file**:

1. Before generating a cross-reference, confirm the heading exists in
   the target file
2. If the target heading is uncertain, use Grep to search for `^#+`
   heading lines in the target file
3. Markdown anchor generation: spaces become `-`, punctuation removed
4. If the most semantically appropriate heading doesn't exist, choose
   the closest parent or related heading — never fabricate anchors

Example (from CODING.md referencing patterns.md):
```markdown
> 🔗 See [Strategy Pattern](arch/patterns.md#strategy-pattern)
```

Example (from SECURITY.md referencing RELIABILITY.md):
```markdown
> 🔗 See [SSO Token Degradation Chain](RELIABILITY.md#sso-token-degradation-chain)
```

## Source Tracing Format

For dimensions with `source_required: true`, each knowledge entry must
include a source marker and scan timestamp:

```markdown
### Strategy Pattern

**Source**: `services/ttadk_cli/src/ai-tools/ai-tool-config.ts:45-78` [Code Direct]
**Scanned**: 2026-04-15

The strategy pattern is used in AIToolConfig for...
```

### Source Marker Types

| Marker | Meaning | Confidence | Use Case |
| --- | --- | --- | --- |
| `[Code Direct]` | Extracted directly from source code with precise file+line range | High | When scanning specific code that can be directly referenced |
| `[AI Inferred]` | AI deduced from code structure, no direct line correspondence | Medium | When inferring design intent from code structure |
| `[External Ref]` | Links to external documentation only | Low | When referencing external API docs or third-party library docs |

### Source Tracing Examples

```markdown
### Factory Pattern

**Source**: `services/ttadk_cli/src/loader/loader-factory.ts:12-45` [Code Direct]
**Scanned**: 2026-04-15

LoaderFactory coordinates the creation of 6 loader types...

---

### Module Decoupling Inference

**Source**: AI Deduction [AI Inferred]
**Scanned**: 2026-04-15

Based on code structure, LoaderFactory's design intent is...
```

### Verification Markers

When Phase 3.5 verification passes, append `[Verified]`:
```markdown
**Source**: `services/ttadk_cli/src/loader/loader-factory.ts:12-45` [Code Direct] [Verified]
```

When verification fails, append `[Pending]`:
```markdown
**Source**: `services/ttadk_cli/src/loader/loader-factory.ts:12-45` [Code Direct] [Pending]
```

## Null-Value Passthrough Rule

If a section's scan results cannot provide incremental value beyond
"reading the source code directly" (e.g., merely restating code
comments, vague descriptions, inferences without concrete evidence),
omit the section entirely — write nothing.

This is the core constraint against AI completion bias (the tendency
to fill content).

**Judgment Criteria**:
- If a reader could get equivalent information by reading source code
  directly → omit
- If a description has conclusions but no evidence → omit
- If content says "the project uses X" without showing how → omit

> All reference document templates follow the same specification as
> patterns.md and CODING.md:
> - Table of contents section (when toc_required=true)
> - Structure field alignment (required=true must produce, required=false
>   only if content exists)

### 12-16. Constitution Documents (CONSTITUTION/QUALITY/RELIABILITY/SECURITY/CODING)

> When a schema.yaml dimension has `output.style: constitution`, use this
> template instead of the generic one.
> Format reference: the corresponding files in
> `templates/constitution/{TYPE}.md`; that directory is the
> sole template source for constitution-style documents.
> Key differences: no TOC section, no source tracing markers, no source
> references section, uses ### I~N numbering instead of ## peer headings.

**Companion Notes**:

1. **Single Template Source**: `templates/constitution/{TYPE}.md`
   defines the constitution document skeleton; `output-templates.md` only
   explains compound write rules
2. **Frontmatter**: Compound-generated results still retain
   `last_compound_scan` + `compound_scan_mode`
3. **Core Format Differences**: No `## Table of Contents`, no source
   tracing markers, Core Section uses `### I.` / `### II.` / `### III.`
   Roman numeral numbering
4. **Remaining Structure**: Fixed Rules / Governance / version line
   paragraph structure and naming follow the corresponding
   constitution/{TYPE}.md files
5. **Null Values and Cross-References**: Empty sections follow the
   global null-value passthrough rule; cross-references still use
   `> 🔗 See [Topic Name](relative-path#anchor)` and follow the
   relative path rules in this document
