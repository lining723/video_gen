# 智能视频生成平台 Constitution

## Core Principles

### I. 产品闭环优先
本项目的所有设计与实现必须服务于“创意输入 → 场景设计 → 分镜审核 → 主体出图 → 镜头渲染 → 最终拼接 → 成片交付”的完整业务闭环。任何新能力若不能明确落入该闭环中的某个阶段，或无法提升交付可用性、可控性、可复用性，则不得优先进入 MVP。所有 `spec.md`、`plan.md`、`tasks.md` 必须能追溯到闭环中的具体业务价值。

### II. 人工审核门禁不可跳过
场景设计审核与分镜审核是本项目的非协商门禁。实现不得绕过 review 环节直接进入下游视频生成；驳回、重新生成、版本保留与审计记录必须作为标准能力存在。任何试图牺牲审核门禁换取短期生成效率的方案，都必须被视为违反宪章。

### III. 契约优先与跨端一致性
前端 Node.js 与后端 Python 的协作必须以清晰、可版本化的接口契约为基础。所有关键业务流都应优先落到显式 API、数据模型和状态字段中，而不是依赖隐式约定。术语、状态名、版本语义、错误码与关键字段在 `README`、`spec.md`、`plan.md`、`contracts/` 和代码实现中必须保持一致。

### IV. 异步媒体流水线可靠性
视频生成、主体出图、拼接等长耗时工作必须通过可追踪的异步任务执行，不得以阻塞式在线请求替代。系统必须支持任务状态跟踪、超时处理、重试、局部失败隔离和缓存降级，以保证镜头级失败不会破坏整体项目推进。任何新实现若不能说明失败恢复路径和任务可观测性，则不得合入。

### V. 可观测性、版本化与简洁实现
本项目必须默认记录项目 ID、镜头 ID、任务 ID、模型版本、缓存命中情况和关键错误摘要，以支撑排查与回溯。场景设计、分镜脚本、主体图与最终成片都必须具备版本化语义。实现上应遵循简洁优先，优先选择能满足当前需求的最小可行方案，避免过度抽象、过早平台化或与当前 MVP 无关的复杂设计。

## 技术与合规约束

- 前端技术栈默认采用 Node.js + TypeScript 体系，后端默认采用 Python 体系；如需偏离，必须在 `plan.md` 中给出理由。
- 任何实现都必须保留缓存目录、对象存储/文件存储、数据库元数据三层职责边界，不得将其混为单一不透明存储。
- 字幕、配音、视频三轨在拼接阶段必须保持时间对齐；涉及该链路的改动必须明确说明验证方式。
- 所有敏感配置、存储密钥、模型访问凭据不得暴露到前端或提交到仓库。
- 非功能目标至少包括：页面关键交互响应、状态查询时延、镜头级并发、失败重试、持久化、告警与审计。

## 开发流程与质量门禁

- 所有新需求必须先进入 `spec.md`，明确用户故事、功能需求、边界情况与成功标准后，才能进入 `plan.md` 和 `tasks.md`。
- `plan.md` 必须说明技术上下文、状态流、数据模型、接口契约、测试策略和主要风险；若缺少这些内容，不得进入任务拆解。
- `tasks.md` 必须显式包含以下类型的工作项：
  - 契约/API 对齐
  - 状态机与版本规则实现
  - 日志/审计/可观测性
  - 失败恢复与重试
  - 缓存命中与失效逻辑
- 涉及状态流、契约、缓存、拼接的改动，必须优先补充或更新对应测试；至少应覆盖单镜头失败、缓存命中、拼接成功/失败等关键路径。
- 任何新增复杂度（例如新增服务层、存储层、任务编排层）都必须在 `plan.md` 的 Complexity Tracking 中说明必要性。

## Fixed Rules

- **提交规则**：当用户通过自然语言请求提交代码（例如“帮我提交”“提交改动”）时，执行 `/adk:commit` 来暂存变更、生成提交信息并按需推送远端。
- **代码一致性**：实现功能时，优先参考现有代码库中的模式、编码风格和架构设计，遵循既有约定，保持一致性。
- **知识检索**：遇到陌生概念时，优先使用 `tiksearch` MCP 检索内部文档与最佳实践，使用 `lark-docs` MCP 读取 Lark/飞书文档。

## Governance

本宪章高于普通实现习惯、临时口头约定和局部文档说明。所有 `spec.md`、`plan.md`、`tasks.md` 及后续实现都必须通过本宪章校验。

宪章修订采用语义化版本：
- **MAJOR**：删除、重定义或破坏既有原则
- **MINOR**：新增原则、门禁或新增强制性章节
- **PATCH**：措辞澄清、示例完善、无语义变化的细化

合规检查要求：
- 在生成或更新 `plan.md` 时，必须执行 Constitution Check。
- 在生成 `tasks.md` 时，必须确认任务覆盖契约、状态机、日志、恢复、缓存等宪章要求。
- 在实施与评审阶段，若发现实现与宪章冲突，应优先修正规格或实现，而不是绕过原则。

修订流程要求：
- 宪章修改必须由用户显式提出，或在治理需求明确时单独执行。
- 每次修订必须更新顶部 Sync Impact Report，并记录版本变化、影响范围与后续待办。
- 若宪章修改影响模板或流程文件，必须在同一轮更新中说明是否同步及原因。
